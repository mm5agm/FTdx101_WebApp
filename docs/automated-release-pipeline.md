📘 Automated Release Pipeline — How It Works
This document explains how the automated release system works after the migration to:

release‑please (automatic versioning + changelog + releases)

WixSharp (modern MSI installer)

GitHub Actions (automated build + upload)

This pipeline removes all manual steps.
Every release is created, versioned, built, and published automatically.

🧭 1. Branch Structure
The project uses a simple, reliable branching model:

main
Always contains production‑ready code

Watched by release‑please

Triggers automated releases

develop
Used for ongoing development

Merged into main via Pull Requests (PRs)

🧭 2. What Happens When You Merge Into main
When a Pull Request is merged into main, the following sequence happens automatically:

Step 1 — release‑please scans the changes
It looks at:

Commit messages

PR titles

Conventional commit keywords (if used)

Then it decides whether to bump:

Major version

Minor version

Patch version

Step 2 — release‑please creates a “release PR”
This PR contains:

A version bump (e.g., 0.6.0 → 0.7.0)

An updated CHANGELOG.md

A Git tag proposal

Release notes

The PR title looks like:

chore: release 0.7.0

Step 3 — You merge the release PR
This is the only manual step in the entire release process.

When you merge it:

GitHub automatically creates a GitHub Release

A version tag is created (e.g., v0.7.0)

The installer build workflow is triggered

🧭 3. What Happens When a Release Is Created
When GitHub creates a release, the workflow:

Code
.github/workflows/build-installer.yml
runs automatically.

This workflow:

✔ Publishes the .NET app
Code
dotnet publish -c Release -r win-x86
✔ Builds the MSI using WixSharp
WixSharp reads:

The version from the release tag

The published output

The installer script

It then generates a proper MSI installer.

✔ Uploads the MSI to the GitHub Release
The installer appears under the release assets automatically.

No manual uploading.
No manual tagging.
No manual building.

🧭 4. Release Flow Summary
Here’s the full pipeline in one view:

Code
develop → PR → main
        ↓
release‑please scans changes
        ↓
release PR created
        ↓
merge release PR
        ↓
GitHub Release created
        ↓
MSI build workflow runs
        ↓
Installer uploaded automatically
This is the complete automated release cycle.

🧭 5. How to Trigger a New Release
You do not manually create tags or releases.

To trigger a new release:

Commit changes to develop

Push develop

Create a PR into main

Merge the PR

Merge the release PR created by release‑please

That’s it.

🧭 6. How Versioning Works
release‑please uses semantic versioning:

fix: → patch version

feat: → minor version

BREAKING CHANGE: → major version

If you don’t use conventional commits, release‑please still works — it just bumps versions based on PR titles and commit patterns.

🧭 7. Where the Installer Comes From
The installer is generated by:

Code
installer/InstallerScript.cs
installer/Installer.csproj
WixSharp handles:

MSI creation

Auto‑upgrade

Program Files installation

Start Menu shortcut

Version injection

The MSI is uploaded to the release automatically.

🧭 8. Manual Actions You No Longer Need
You no longer need to:

Manually create tags

Manually create GitHub Releases

Manually upload installers

Manually run NSIS

Manually bump versions

Manually write changelogs

All of this is automated.

🧭 9. Troubleshooting
No release PR appears
Make sure the merge went into main.

Installer doesn’t appear
Check the workflow:

Code
Actions → Build MSI Installer
Version looks wrong
release‑please controls the version — don’t manually edit it.
