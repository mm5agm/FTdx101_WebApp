# Architecture Maturity Model
Defines stages of architectural maturity for the FTdx101 WebApp.
## Level 1: Ad‑hoc
No boundaries; logic mixed across files; no tests; UI and logic intertwined; no documentation.
## Level 2: Structured
Basic folders; some separation; inconsistent patterns; partial documentation; minimal tests; drift risk high.
## Level 3: Modular
Clear subsystems; pipeline mostly stable; tests for core logic; UI isolated; documentation present but incomplete.
## Level 4: Defined
Strict subsystem boundaries; calibration/formatting/UI/WebSocket separated; pipeline enforced; drift-check used; onboarding docs complete; consistent coding standards.
## Level 5: Optimised
Performance guidelines applied; tests cover all logic; UI predictable; subsystems evolve safely; roadmap maintained; accessibility and theming defined; architecture stable under change.
## Level 6: Self‑Sustaining
Architecture enforces itself; drift detection automated; subsystem evolution predictable; contributors onboard quickly; performance, accessibility, and theming integrated; long‑term roadmap active.
## Usage
Use maturity levels to assess current state, plan improvements, and prevent regression.
## Summary
Maturity increases as boundaries strengthen, documentation grows, tests expand, and the pipeline remains stable under evolution.
