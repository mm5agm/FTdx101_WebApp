# Refactor Session

This file defines how Copilot should perform a refactor session within this project.  
All actions must follow the project-wide rules defined in `.github/copilot-instructions.md`.

## Purpose
A refactor session applies the project’s refactor rules to a specific set of files.  
Copilot must produce coherent refactor chunks, each with:
- A one- or two-sentence summary of the refactor’s drivers.
- Standalone rationales for each change.
- Rationales grouped by subsystem.
- Subsystems ordered dynamically based on which areas were touched.
- Natural, human phrasing.

## Instructions for Copilot
When this file is invoked with a list of files:

1. Analyse the provided files for:
   - Architectural drift
   - Naming drift
   - Duplicated logic
   - Outdated comments
   - Outdated support messaging
   - Folder/namespace inconsistencies
   - Opportunities for helper extraction

2. Apply the project’s refactor rules:
   - Consolidate duplicated logic aggressively.
   - Extract shared behaviour into new helper methods/classes.
   - Reorganise helpers into appropriate folders/namespaces.
   - Rename folders/namespaces when conceptual boundaries have evolved.
   - Use camelCase with longer descriptive names for all identifiers, including private fields.
   - Update comments to reflect current behaviour.
   - Align support messaging with internal behaviour.

3. Produce output in the required refactor-chunk format:
   - Start with a summary sentence (one or two sentences).
   - Group standalone rationales by subsystem.
   - Order subsystem groups dynamically.
   - Use natural, concise language.

4. Ensure all changes maintain architectural clarity and reflect empirical behaviour.
